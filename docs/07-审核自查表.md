# 最佳干饭人 · 审核被拒原因自查表

> 个人主体小程序审核高发问题清单 + 自查方法。建议在提交审核前完整过一遍。

---

## ⚠️ 总体原则

**个人主体**小程序审核比企业主体**严格**：
- 不能涉及社交、社区、群组
- 不能涉及支付、虚拟商品
- 不能涉及医疗建议、健康咨询
- 不能涉及新闻、时政
- 必须有完整的用户协议和隐私政策

---

## 一、类目相关（被拒率 60%）

### 🔴 高危：选错类目

| 错误选择 | 必拒原因 | 正确选择 |
|---|---|---|
| 社交 | 涉及 1:1 关系被判定社交 | ❌ 严禁 |
| 社区 | 任何"社区"字样都拒 | ❌ 严禁 |
| 工具 → 效率 | 部分审核员要求证明 | ⚠️ 备选 |
| 工具 → 健康管理 | **最合规** | ✅ 推荐 |
| 医疗 → 医疗服务 | 需要医疗资质 | ❌ 个人主体不可能 |

**自查方法**：
- [ ] 类目 = "**工具 → 健康管理**"
- [ ] 类目描述里**没有**"社交""社区""群组""动态"等词
- [ ] 类目描述强调"**个人记录工具**"

### 🟡 中危：描述措辞

```
❌ 错误示例：
"健康饮食工具，帮助用户养成健康饮食习惯"
"营养管理神器，监督你按时吃饭"
"和朋友一起打卡，互相督促"

✅ 正确示例：
"个人饮食记录工具，支持 1:1 私密数据分享"
"为用户提供饮食打卡功能，不提供医疗建议"
"1:1 配对的记录工具，非公开社区"
```

**自查方法**：
- [ ] 描述中只用"**记录**""**工具**""**搭子**""**配对**"等中性词
- [ ] 没有"**健康饮食**""**监督**""**督促**"等可能被判医疗/社交的词

## 二、隐私合规（被拒率 25%）

### 🔴 高危：缺失协议

**必查**：
- [ ] `app.json` 中有隐私相关弹窗配置
- [ ] 用户**首次**使用时必须弹隐私政策，用户**主动勾选**后才能继续
- [ ] 协议内容**有"最后更新日期"**
- [ ] 协议里有"**联系方式**"（邮箱或微信）

### 🟡 中危：协议内容不全

**《用户协议》必含**：
- [ ] 服务说明
- [ ] 用户行为规范
- [ ] 内容审核（提及 `imgSecCheck`/`textSecCheck`）
- [ ] 隐私保护（指向隐私政策）
- [ ] 免责条款
- [ ] 协议变更说明
- [ ] 联系方式

**《隐私政策》必含**：
- [ ] 收集哪些信息（具体到字段）
- [ ] 收集目的
- [ ] 存储位置、期限
- [ ] 共享场景
- [ ] 用户权利（查看/删除/注销）
- [ ] 第三方服务（云开发、内容安全）
- [ ] 政策更新机制
- [ ] 联系方式

### 🔴 高危：强制授权

**自查方法**：
- [ ] 微信授权按钮**不强制**点击
- [ ] 提供"**拒绝**"入口
- [ ] 拒绝后**仍可使用**部分功能（或明确告知无法使用）

---

## 三、内容安全（被拒率 10%）

### 🔴 高危：未做内容检测

**自查方法**：
- [ ] 图片上传**强制**经过 `imgSecCheck`
- [ ] 文字输入**强制**经过 `textSecCheck`
- [ ] 检测失败的内容**拒绝保存**
- [ ] 违规内容**不向其他用户展示**

### 🟡 中危：UGC 留存

- [ ] 内容审核日志**至少留存 60 天**（云函数内 print 日志即可）

---

## 四、敏感权限（被拒率 5%）

### 🔴 高危：申请多余权限

**自查方法**：
- [ ] `app.json` 中 `permission` 字段**只申请真正用到的权限**

| 权限 | 是否需要 | 说明 |
|---|---|---|
| `scope.userLocation` | ❌ 不需要 | 我们的 MVP 不使用位置 |
| `scope.userAddress` | ❌ 不需要 | |
| `scope.address` | ❌ 不需要 | |
| `scope.record` | ❌ 不需要 | |
| `scope.album` | ✅ 需要 | 拍照/选图 |
| `scope.camera` | ✅ 需要 | 拍照 |
| `scope.invoiceTitle` | ❌ 不需要 | |
| `scope.invoice` | ❌ 不需要 | |
| `scope.userInfo` | ✅ 需要 | 头像昵称 |

**当前 app.json 应移除**：
```json
"permission": {
  "scope.userLocation": {
    "desc": "用于显示附近餐厅"  // ← 我们没用，删掉！
  }
}
```

### 当前 app.json 已修复（task 2 之后）

## 五、功能完整性（被拒率 < 5%）

### 🟡 中危：核心功能不可用

- [ ] 至少 5 个核心页面**能正常打开**
- [ ] 主流程**能跑通**：登录 → 拍照 → 提交 → 查看记录
- [ ] 没有"**TODO**""**开发中**""**404**"等占位

### 🔴 高危：空页面/404

**自查方法**：
- [ ] `app.json` 中 `pages` 数组的**每个页面都有 4 个文件**（.wxml/.wxss/.js/.json）
- [ ] 没有空目录
- [ ] `tabBar.list` 中**每个 pagePath 都在 pages 数组里**

| 新闻 | 头条、新闻、资讯、时政 |

| 色情 | （完全禁用） |
| 赌博 | （完全禁用） |

**自查方法**：
- [ ] 在项目根目录执行：
  ```bash

  ```
- [ ] 检查输出，把命中行逐一审查

## 🆘 被拒后处理流程

```
审核未通过
    ↓
1. mp 后台查看具体原因（"理由"字段）
    ↓
2. 对照本表定位问题
    ↓
3. 修改代码 / 描述
    ↓
4. 重新上传体验版
    ↓
5. 重新提交审核
    ↓
6. 继续
```

### 申诉技巧

如果觉得被拒原因不合理：
1. mp 后台 → 客服中心
2. 选择"小程序审核"类目
3. 提供详细说明和截图
4. 引用具体协议条款作为证据
